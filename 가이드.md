# AI 활용 가이드

이 문서는 AI가 프로젝트의 요구사항과 안전 규칙을 지키면서 일관되게 작업하도록 사용하는 방법을 설명합니다.

## 1. 기본 원칙

AI에게 단순히 “기능을 만들어줘”라고 요청하기보다 다음 정보를 함께 전달합니다.

1. 무엇을 만들거나 수정할지
2. 완료 조건이 무엇인지
3. 수정해도 되는 범위
4. 참고해야 할 이슈·Notion·API 문서
5. 실행해야 할 테스트

권장 요청 형식:

```text
[작업 목적]을 구현해줘.

- 요구사항: [구체적인 기능]
- 완료 조건: [사용자에게 보여야 하는 결과]
- 작업 범위: [frontend 또는 backend 경로]
- 참고 문서: [이슈·Notion·API 문서 링크]
- 검증: 관련 test, lint, build 실행

완료 후 변경 파일, 검증 결과, 남은 위험을 정리해줘.
```

## 2. AGENTS.md

`AGENTS.md`는 AI가 모든 작업에서 지켜야 하는 상시 규칙입니다.

```text
AGENTS.md             공통 작업 순서, 권한, 보안, 검증 규칙
frontend/AGENTS.md    Vue 프론트엔드 전용 규칙
backend/AGENTS.md     Spring Boot 백엔드 전용 규칙
```

AI에게 작업을 요청할 때 다음과 같이 말합니다.

```text
루트와 작업 대상 폴더의 AGENTS.md를 먼저 읽고 규칙에 따라 작업해줘.
```

Codex처럼 `AGENTS.md`를 지원하는 도구는 작업 범위에 해당하는 규칙을 자동으로 확인할 수 있습니다. 다른 AI 도구에서는 위 문장을 요청에 명시합니다.

### AGENTS.md에 두기 좋은 내용

- 항상 지켜야 하는 보안·권한 규칙
- 수정 가능·승인 필요·수정 금지 영역
- 프로젝트 구조와 공통 API 계약
- 기본 test, lint, build 명령
- 완료 보고에 포함해야 할 항목

특정 작업에서만 필요한 긴 절차는 `AGENTS.md`가 아니라 Skill이나 `docs/`에 둡니다.

## 3. Skill 사용법

Skill은 반복 작업의 순서와 완료 기준을 저장한 작업 절차입니다. 같은 절차를 Codex와 Claude가 각각 자동 발견할 수 있는 위치에 두었습니다.

| Skill | 사용 시점 | Codex 위치 | Claude 위치 |
| --- | --- | --- | --- |
| `backend-issue-workflow` | 백엔드 기능·버그 구현 | `.agents/skills/backend-issue-workflow/` | `.claude/skills/backend-issue-workflow/` |
| `frontend-issue-workflow` | 프론트엔드 기능·버그 구현 | `.agents/skills/frontend-issue-workflow/` | `.claude/skills/frontend-issue-workflow/` |
| `pr-check` | PR/MR 생성 전 최종 검사 | `.agents/skills/pr-check/` | `.claude/skills/pr-check/` |

Codex는 현재 작업 폴더부터 저장소 루트까지의 `.agents/skills`를 검색합니다. Skill 파일이 변경되면 자동 감지하며, 목록에 나타나지 않으면 Codex를 다시 시작하거나 새 작업을 엽니다.

Skill을 지원하는 도구에서는 이름을 직접 지정합니다.

```text
$backend-issue-workflow을 사용해서 회원 조회 API를 구현해줘.
```

```text
$frontend-issue-workflow을 사용해서 회원 조회 화면을 구현해줘.
```

```text
$pr-check을 사용해서 현재 변경 사항을 점검하고 PR 본문을 작성해줘.
```

도구가 프로젝트 Skill을 자동으로 찾지 못하면 사용하는 도구의 파일 경로를 함께 전달합니다.

```text
Codex에서는 .agents/skills/backend-issue-workflow/SKILL.md를 읽고 그 절차에 따라 작업해줘.
Claude에서는 .claude/skills/backend-issue-workflow/SKILL.md를 읽고 그 절차에 따라 작업해줘.
```

### 새 Skill이 필요한 경우

다음 조건에 해당하면 별도 Skill로 분리합니다.

- 같은 작업 절차를 세 번 이상 반복하는 경우
- 작업 순서가 빠지면 보안이나 품질 문제가 발생하는 경우
- 입력, 실행 순서, 산출물, 완료 조건이 명확한 경우

Skill에는 목적, 먼저 읽을 근거, 수정 범위, 작업 순서, 검증 명령, 산출물, 금지사항과 완료 기준을 작성합니다.

## 4. Hook 사용법

Hook은 AI나 Git 작업 전후에 자동으로 실행되는 검사입니다.

### Codex Hook

Codex 설정은 `.codex/config.toml`, Hook 정의는 `.codex/hooks.json`에 있습니다.

| 시점 | 실행 파일 | 역할 |
| --- | --- | --- |
| 도구 실행 전 | `.codex/hooks/guard.sh`, `guard.ps1` | Secret 파일과 위험 명령 차단 |
| 파일 수정 후 | `.codex/hooks/quick-check.sh`, `quick-check.ps1` | 변경 파일의 하드코딩된 Secret 검사 |

- macOS·Linux에서는 `.sh`, Windows에서는 `.ps1` 파일을 실행합니다.
- 프로젝트를 처음 열거나 Hook이 변경되면 Codex의 `/hooks`에서 내용을 검토하고 신뢰해야 실행됩니다.
- `.codex/` 프로젝트 설정은 신뢰된 프로젝트에서만 로드됩니다.
- Hook은 일부 도구 호출을 가로채는 안전장치이며 완전한 보안 경계가 아닙니다. Git Hook과 Jenkins 검사를 함께 유지합니다.

Codex에서 다음과 같이 상태를 확인합니다.

```text
/hooks
```

### Claude Hook

Claude Hook 설정은 `.claude/settings.json`에 있습니다.

| 시점 | 실행 파일 | 역할 |
| --- | --- | --- |
| 작업 실행 전 | `.claude/hooks/guard.sh` | Secret 파일과 위험 명령 차단 |
| 파일 수정 후 | `.claude/hooks/quick-check.sh` | 하드코딩된 Secret 의심 값 검사 |
| 작업 종료 전 | `.claude/hooks/run-related-tests.sh` | 변경 영역의 최소 검증 실행 |

차단 대상 예시:

- 실제 `.env`, PEM, private key 읽기·수정
- `git reset --hard`와 force push
- 광범위 재귀 삭제
- 운영 DB drop
- 승인되지 않은 Kubernetes 리소스 삭제

Hook이 작업을 차단하면 AI에게 우회하도록 지시하지 않습니다. 차단 이유를 확인하고 필요한 경우 팀원의 승인을 받은 뒤 안전한 방법으로 진행합니다.

Claude의 `.sh` Hook은 셸 환경이 필요합니다. Windows에서는 Git Bash를 사용할 수 있습니다.

### Git Hook

Git Hook은 개발자가 직접 커밋하거나 push할 때도 동일한 검사를 적용합니다.

저장소를 Git으로 초기화한 뒤 한 번 실행합니다.

```powershell
git config core.hooksPath .githooks
```

- `pre-commit`: 실제 환경 파일, 키 파일, 하드코딩된 Secret 검사
- `commit-msg`: `<gitmoji> <type>: <명령형 요약>` 커밋 제목 검사
- `pre-push`: 프론트 lint/test/build와 백엔드 test/build 실행

커밋 제목은 다음 Gitmoji 규칙을 사용합니다.

```text
✨ feat: 사용자 기능 또는 API 추가
🐛 fix: 버그 수정
♻️ refactor: 외부 동작을 바꾸지 않는 구조 개선
✅ test: 테스트 추가·정리
📝 docs: 문서 변경
🚀 infra: Docker, Jenkins, Kubernetes, 배포 변경
🔧 chore: 도구·의존성·단순 유지보수
⚡️ perf: 성능 개선
🔒 security: 보안 개선
```

Hook을 통과하지 못한 변경은 원인을 해결한 뒤 다시 실행합니다. `--no-verify`를 기본 해결책으로 사용하지 않습니다.

## 5. Notion과 외부 도구 사용

Notion, 이슈 관리 도구, Git 저장소가 AI에 연결되어 있으면 문서를 복사해 붙이는 대신 원문을 직접 조회하도록 요청할 수 있습니다.

```text
Notion의 API 명세와 연결된 이슈를 먼저 확인해줘.
관련된 부분만 읽고 코드와 계약이 다르면 구현하지 말고 차이를 알려줘.
```

외부 도구는 다음 권한 원칙을 적용합니다.

- 조회 권한부터 사용합니다.
- 댓글, 문서 수정, PR 생성은 요청받은 범위에서만 수행합니다.
- 운영 배포, DB 쓰기, Secret 변경은 별도 승인 없이 실행하지 않습니다.
- 확인한 요구사항은 원문 링크와 함께 결과에 남깁니다.
- 연결 실패 시 기억에 의존해 완료 처리하지 않습니다.

## 6. 작업 유형별 요청 예시

### 백엔드 기능 구현

```text
$backend-issue-workflow을 사용해 `/api/v1/members/{id}` 조회 API를 구현해줘.

- 루트와 backend/AGENTS.md를 먼저 확인
- 공통 ApiResponse 사용
- 정상 조회와 존재하지 않는 회원 테스트 추가
- API 문서 갱신
- gradle test와 bootJar 실행

운영 설정과 DB migration은 변경하지 마.
```

### 프론트엔드 기능 구현

```text
$frontend-issue-workflow을 사용해 회원 상세 화면을 구현해줘.

- 로딩, 성공, 오류 상태 표시
- API 주소 하드코딩 금지
- 사용자 동작 테스트 추가
- lint, test, build 실행
```

### 버그 진단만 요청

```text
현재 오류의 원인만 진단해줘.
코드는 수정하지 말고 재현 조건, 원인, 영향 범위와 권장 수정 방법을 정리해줘.
```

### PR 최종 점검

```text
$pr-check을 사용해 현재 diff를 확인해줘.
Secret, 변경 범위, API 문서, 테스트, 빌드, 배포와 rollback 위험을 점검하고 GitLab MR 양식으로 요약해줘.
```

## 7. 권장 작업 흐름

```text
작업 요청
→ AGENTS.md와 해당 Skill 확인
→ 이슈·Notion 요구사항 확인
→ 관련 코드와 인접 테스트만 확인
→ Hook 정책 안에서 구현
→ lint·test·build
→ PR 점검
→ 사람 리뷰
→ Jenkins 최종 검증
→ 승인 후 배포
```

빠른 검사는 로컬 Hook에서 실행하고, 전체 테스트·JaCoCo·SonarQube·Docker 빌드는 Jenkins에서 최종 확인합니다.

## 8. AI에게 맡기면 안 되는 작업

명시적인 승인과 사람의 확인 없이 다음 작업을 맡기지 않습니다.

- 운영 배포와 Rollback 실행
- 운영 DB 데이터 변경·삭제
- AWS IAM, Jenkins credential, Kubernetes Secret 변경
- 실제 Secret, 토큰, 개인정보의 조회 또는 출력
- force push, 기록 삭제, 광범위 파일 삭제
- 실패한 테스트나 Quality Gate 우회

AI가 생성한 코드와 설명도 틀릴 수 있으므로 최종 merge와 배포 판단은 사람이 담당합니다.

## 9. 완료 결과 확인표

AI가 작업을 완료했다고 하면 다음 항목을 확인합니다.

- [ ] 요구사항과 완료 조건을 충족했는가?
- [ ] 요청 범위 밖 파일을 수정하지 않았는가?
- [ ] Secret이나 개인정보가 포함되지 않았는가?
- [ ] API·DB 계약 변경에 테스트와 문서가 포함됐는가?
- [ ] 관련 lint, test, build가 통과했는가?
- [ ] 실행하지 못한 검증을 명확하게 밝혔는가?
- [ ] 배포 영향과 Rollback 방법이 필요한 경우 작성됐는가?
- [ ] 사람 리뷰가 필요한 판단이 구분되어 있는가?

## 10. 문제 해결

### Skill을 찾지 못하는 경우

- Codex: 새 작업을 열거나 다시 시작한 뒤 `.agents/skills/<skill-name>/SKILL.md` 경로를 명시합니다.
- Claude: `.claude/skills/<skill-name>/SKILL.md` 경로를 명시합니다.

### Codex Hook이 실행되지 않는 경우

1. 프로젝트가 신뢰된 상태인지 확인합니다.
2. Codex에서 `/hooks`를 열어 새 Hook을 검토하고 신뢰합니다.
3. `.codex/config.toml`의 `[features] hooks = true`를 확인합니다.
4. Windows에서는 PowerShell 실행 정책 오류가 있는지 확인합니다.

### 프론트 검증이 실행되지 않는 경우

```powershell
cd frontend
npm install
npm run lint
npm run test
npm run build
```

### 백엔드 검증이 실행되지 않는 경우

Java 21과 Gradle 8.14 이상 또는 Gradle 9가 필요합니다.

```powershell
cd backend
gradle test
gradle bootJar
```

### Hook이 정상 파일을 차단하는 경우

Hook을 끄기 전에 차단된 파일과 메시지를 확인합니다. 오탐이라면 팀 검토 후 Hook 패턴을 수정하고, 수정 이유와 검증 결과를 함께 기록합니다.
